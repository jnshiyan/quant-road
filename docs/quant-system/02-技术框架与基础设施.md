> 版本：V1.0  
> 日期：2026-05-05  
> 主题：说明这套量化系统当前依赖什么技术框架、真实执行链路是什么、哪些地方已经落地、哪些地方还处于目标状态

---

## 1. 为什么要单独写这一章

量化系统最容易出现的一个问题，是“名义上的技术栈”和“真实运行的执行链路”并不完全一致。

例如：

1. 文档里可能写了某个回测框架。
2. 但系统实际每天跑的逻辑，可能更多是项目内部封装的服务函数。

如果这一层不讲清楚，后面关于回测、策略、任务、页面的讨论都会漂浮。

因此，这一章的目标是：

1. 解释当前系统的技术分层。
2. 明确“框架是什么”和“实际是谁在运行”。
3. 为后续策略、风控、回测、任务治理文档提供统一技术口径。

## 2. 系统总体分层

当前系统可以分成四层。

### 2.1 Java 管控层

主要目录：

1. `ruoyi-admin`
2. `ruoyi-ui`
3. `ruoyi-quartz`

主要职责：

1. 登录、权限、菜单
2. 后台接口
3. 任务触发
4. 页面展示
5. 定时任务接入
6. Python 子进程调用的控制入口

业务理解：

这层更像“控制台”和“调度台”，而不是量化策略本身。

### 2.2 Python 量化层

主要目录：

1. `python/src/quant_road/collectors`
2. `python/src/quant_road/services`
3. `python/src/quant_road/strategies`

主要职责：

1. 数据采集
2. 市场状态判断
3. 回测与信号生成
4. 风控与持仓更新
5. 执行反馈
6. 报告生成
7. 异步任务 worker 执行

业务理解：

这层才是量化系统的业务平面，真正的投资规则主要在这里。

### 2.3 数据层

当前核心为 PostgreSQL。

主要职责：

1. 股票基础信息存储
2. 日线行情存储
3. 市场状态与估值快照存储
4. 策略配置存储
5. 信号、持仓、成交、反馈、日志存储
6. 异步任务和批次状态存储

业务理解：

PostgreSQL 在当前系统中不是单纯缓存，而是**业务事实库**。

### 2.4 协调层

当前核心为 Redis。

主要职责：

1. 异步任务 shard 排队
2. worker 租约管理
3. 失败恢复与重入协调

业务理解：

Redis 不是量化数据源，而是长任务系统的协调组件。

## 3. 当前基础技术框架清单

## 3.1 数据采集框架

### 当前实现

当前主要数据采集框架是：

1. `AKShare`

当前用途包括：

1. 股票基础信息采集
2. 股票日线数据采集
3. 指数估值数据采集
4. 市场状态相关指数数据采集

代码落点：

1. `python/src/quant_road/collectors/akshare_collector.py`
2. `python/src/quant_road/services/valuation_service.py`
3. `python/src/quant_road/services/market_regime_service.py`
4. `python/src/quant_road/services/risk_service.py`

### 技术判断

AKShare 在当前系统中承担的是“统一外部市场数据入口”的角色。

优点：

1. A股数据覆盖广
2. 接口丰富
3. 快速落地成本低

限制：

1. 上游接口可能不稳定
2. 不同接口之间字段口径可能不完全一致
3. 在估值和指数类数据上，当前系统已出现多源回退需求

### 目标标准

后续成熟形态应做到：

1. 对采集源做统一封装，而不是业务层直接感知具体 AKShare 接口差异
2. 明确每种数据的主源和备源
3. 关键数据采集失败时，有稳定 fallback 策略和清晰日志

## 3.2 回测框架

这是当前系统最需要讲清楚的一层。

### 当前实现

从依赖角度看，项目声明了：

1. `Backtrader`

但从当前代码执行链路看，日常策略执行和批量结果生成并不是简单把所有逻辑直接交给 Backtrader 外部引擎完成，而是通过项目内部封装的策略执行服务完成：

1. `python/src/quant_road/strategies/registry.py`
2. `python/src/quant_road/services/strategy_execution_service.py`
3. `python/src/quant_road/services/strategy_service.py`

也就是说，当前口径应明确区分两层：

1. **依赖层**
   - 项目有 `Backtrader` 依赖
2. **实际执行层**
   - 当前业务输出主要由项目内部策略执行器与回测指标结构驱动

### 技术判断

这意味着当前系统更接近：

1. 在项目内部固化了自己的策略执行契约
2. 再结合现有回测能力输出统一指标

这样做的优点是：

1. 更容易把信号生成、运行日志、风控规则、异步批处理统一起来
2. 更容易做针对当前业务的定制指标输出

限制是：

1. 如果文档只写“回测框架是 Backtrader”，会掩盖真实执行链路
2. 后续需要更明确地整理“依赖框架”和“业务执行器”之间的关系

### 目标标准

成熟形态建议明确成：

1. 统一的策略执行接口
2. 统一的指标输出结构
3. 统一的交易成本模型
4. 统一的回测结果持久化方式
5. 明确说明哪些能力由外部框架提供，哪些能力由项目内部封装提供

## 3.3 Java 管控框架

### 当前实现

当前 Java 管控框架基于：

1. `RuoYi-Vue`

它承担的主要不是量化计算，而是后台管理能力：

1. 菜单与权限
2. REST 接口
3. 任务触发
4. 定时任务入口
5. 页面展示

当前量化页面已经接入：

1. `量化看板`
2. `影子对比`
3. `任务中心`
4. `执行回写`

### 技术判断

RuoYi-Vue 在当前项目中承担的是成熟后台壳层角色。

优点：

1. 权限和菜单能力现成
2. 任务管理、接口管理、后台页面落地快
3. 适合作为量化运营控制台

限制：

1. 它不是量化框架
2. 业务复杂度上升后，页面信息架构和权限分层仍需产品化整理

### 目标标准

成熟形态应保持：

1. Java 层做控制平面
2. Python 层做业务平面
3. 不把量化规则重新复制到 Java 层

## 3.4 任务调度框架

### 当前实现

当前任务调度主要依赖：

1. `RuoYi Quartz`
2. Java 调用 Python CLI

已有任务 Bean：

1. `quantRoadTask`

它可以触发：

1. `fullDaily`
2. `syncBasic`
3. `syncDaily`
4. `syncValuation`
5. `evaluateMarket`
6. `runStrategy`
7. `runPortfolio`
8. `evaluateRisk`
9. `evaluateExecutionFeedback`
10. `notifySignals`
11. `monthlyReport`
12. `shadowCompare`
13. `canaryEvaluate`

### 技术判断

当前系统不是独立重造调度平台，而是选择：

1. 用若依现成调度能力做入口
2. 用 Python 命令承接实际量化执行

这是合理的，因为量化系统当前真正的复杂度在业务链路，不在调度器本身。

### 目标标准

成熟形态需要补强的是：

1. 调度入口与长任务治理的协同
2. 同步与异步模式的透明切换
3. 失败恢复与结果可视化

## 3.5 存储框架

### 当前实现

当前核心存储为：

1. `PostgreSQL`

它存的不只是原始数据，还包括：

1. 股票基础信息
2. 股票日线
3. 市场状态
4. 指数估值
5. 策略配置
6. 信号
7. 持仓
8. 成交
9. 执行反馈
10. 策略运行日志
11. 策略切换审计
12. canary 结论
13. 流水线批次与步骤
14. 异步任务、分片、结果

### 技术判断

PostgreSQL 在这里不是一个简单关系库，而是：

1. 统一量化事实库
2. 统一运维审计库
3. 统一任务状态库

### 目标标准

成熟形态建议保持这一定位，不建议把关键事实层做成临时缓存或分散在多个文件里。

## 3.6 异步协调框架

### 当前实现

当前异步任务协调使用：

1. `Redis`

它主要支持：

1. shard 排队
2. worker lease
3. expired shard recover
4. 失败重试

代码落点：

1. `python/src/quant_road/services/async_job_service.py`
2. `python/src/quant_road/services/async_worker_service.py`
3. `ruoyi-admin/src/main/java/com/ruoyi/web/service/quant/QuantJobPlannerService.java`

### 技术判断

这说明系统已经承认一件现实：

1. 有些量化任务不是短平快请求
2. 必须用 job / shard / worker 模型承接

### 目标标准

成熟形态应继续强化：

1. 任务状态透明化
2. shard 结果聚合
3. worker 故障恢复
4. 用户可理解的错误解释

## 4. 当前真实执行链路

为了避免“框架名很多，但不知道谁在干活”，这里用业务语言描述一次真实链路。

### 4.1 页面与任务入口

用户通过以下入口发起动作：

1. 若依页面
2. 若依定时任务
3. Python CLI

### 4.2 Java 层做的事

Java 层负责：

1. 接收请求
2. 做权限控制
3. 决定调用哪个 Python 命令或 job 规划逻辑
4. 返回结果或任务状态

### 4.3 Python 层做的事

Python 层负责：

1. 采集数据
2. 读取策略配置
3. 加载历史行情
4. 执行回测与信号生成
5. 落库信号、日志、持仓、反馈、任务结果

### 4.4 PostgreSQL 做的事

PostgreSQL 负责：

1. 保存数据事实
2. 保存业务结果
3. 保存执行状态
4. 供页面查询与复盘

### 4.5 Redis 做的事

Redis 负责：

1. 暂存异步任务队列
2. 协调 worker 执行长任务分片

## 5. 当前实现与目标标准

## 5.1 当前实现

当前技术框架层面，已经比较明确的能力包括：

1. AKShare 作为主采集框架
2. RuoYi-Vue 作为控制台与调度入口
3. PostgreSQL 作为统一事实库
4. Redis 作为异步任务协调组件
5. Python 内部服务作为日常量化执行主链路

### 当前最需要诚实指出的点

1. “回测框架”不能只写 Backtrader，需要同时说明项目内部实际执行链路
2. “量化系统”与“后台框架”边界要持续保持清晰
3. 文档中所有框架描述都必须落回真实职责，而不是只列依赖名

## 5.2 目标标准

理想状态下，这一层应达到：

1. 外部框架角色定义清晰
2. 内部执行契约定义清晰
3. 数据采集、回测、策略、任务治理的边界清晰
4. 任何读者都能回答：
   - 谁采集数据
   - 谁执行回测
   - 谁生成信号
   - 谁触发任务
   - 谁持久化事实
   - 谁协调异步
