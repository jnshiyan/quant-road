# 页面体系与操作流程说明

> 版本：V1.0  
> 日期：2026-05-05  
> 主题：用业务语言说明当前量化页面分别怎么用、先看什么、后做什么、每个页面的输入输出和完成标准是什么

---

## 1. 为什么要单独写这一章

用户最早的直接反馈就是：

1. 页面不少
2. 名称看着像能做很多事
3. 但不知道实际应该怎么用

这说明当前系统最大的问题之一，不只是功能有没有，而是：

1. 页面和业务动作没有被明确绑定
2. 页面顺序和日常操作顺序没有被明确说明

所以这一章不再解释技术实现，而是回答：

1. 用户每天进入系统先看哪里
2. 不同页面分别解决什么问题
3. 每个页面的输入、输出、完成标准是什么
4. 页面之间如何形成一条连续操作链

---

## 2. 页面体系先讲结论

当前量化系统的页面，不应理解为四个平行菜单，而应理解为一条盘后运营链路上的四个工作台：

1. `量化看板`
   - 看今天发生了什么
2. `任务中心`
   - 让系统把今天该跑的流程跑起来
3. `执行回写`
   - 把真实成交写回系统
4. `影子对比`
   - 决定策略该不该换

如果按时间顺序理解，这四页的关系更接近：

1. 任务中心先跑
2. 量化看板查看结果
3. 次日执行后进入执行回写
4. 周期性进入影子对比做策略治理

---

## 3. 用户角色与页面分工

## 3.1 量化运营

最常用页面：

1. 任务中心
2. 量化看板
3. 执行回写

主要职责：

1. 跑盘后流程
2. 看当日信号和风险
3. 回写成交
4. 检查执行反馈

## 3.2 策略研究

最常用页面：

1. 量化看板
2. 影子对比
3. 复盘页面

主要职责：

1. 看策略行为
2. 看回测和治理结果
3. 判断策略是否观察、替换

## 3.3 值班排障 / 研发

最常用页面：

1. 任务中心
2. 量化看板
3. 执行回写

主要职责：

1. 看流程有没有失败
2. 看失败发生在哪一步
3. 做重试、恢复、核对

---

## 4. 标准日常使用顺序

## 4.1 盘后标准顺序

盘后建议按以下顺序操作：

1. 进入任务中心，触发 `full-daily` 或确认定时任务已完成
2. 打开量化看板，查看市场状态、当日信号、当前持仓、风险提示
3. 如果任务异常，回到任务中心定位失败步骤或异步分片
4. 输出次日执行清单

## 4.2 次日执行顺序

次日建议按以下顺序操作：

1. 按前一日清单执行真实交易
2. 进入执行回写，录入或导入成交
3. 检查持仓是否同步、信号是否标记执行
4. 检查执行反馈是否仍有 pending / missed

## 4.3 周 / 月治理顺序

建议按以下顺序操作：

1. 打开复盘页面看典型交易和净值行为
2. 打开影子对比看候选策略与基线策略表现
3. 形成观察、保留、替换结论

## 5. 页面一：量化看板

## 5.1 页面定位

量化看板是：

**盘后总览页，不是盘中交易终端。**

## 5.2 页面回答的问题

用户进入这一页，应该能在几分钟内回答：

1. 今天市场环境如何
2. 今天系统产生了哪些新信号
3. 当前有哪些持仓和风险提示
4. 最近策略运行是否异常

## 5.3 页面输入

主要来自：

1. 市场状态
2. 指数估值快照
3. 当日信号
4. 当前持仓
5. 最近策略日志

## 5.4 页面输出

用户离开页面前，至少应形成：

1. 是否需要次日执行 BUY / SELL
2. 是否存在持仓风险需要重点关注
3. 是否存在策略异常需要进一步复盘

## 5.5 推荐模块结构

建议按以下顺序展示：

1. 顶部总览卡
2. 市场状态与估值卡
3. 当日信号表
4. 当前持仓表
5. 风险预警表
6. 最近策略日志

## 5.6 当前实现

当前已基本具备：

1. 市场状态
2. 估值快照
3. 信号
4. 持仓
5. 策略日志

## 5.7 当前短板

仍缺：

1. 信号解释图
2. 净值与回撤图
3. 风险原因解释

## 5.8 页面完成标准

一个合格的看板页，不是信息很多，而是用户能在 3 到 5 分钟内完成：

1. 看市场
2. 看信号
3. 看仓位
4. 看风险

这四件事。

---

## 6. 页面二：任务中心

## 6.1 页面定位

任务中心是：

**盘后生产控制台 + 值班排障台。**

## 6.2 页面回答的问题

用户进入这一页，应该能回答：

1. 今天盘后主流程跑了没有
2. 跑到哪一步
3. 失败在哪里
4. 能不能恢复

## 6.3 页面输入

主要来自：

1. job 批次
2. step 明细
3. async job 状态
4. shard 状态
5. job result

## 6.4 页面输出

用户离开页面前，至少应形成：

1. 当前流程是否完成
2. 是否需要重试
3. 失败原因大致属于哪一类

## 6.5 推荐模块结构

建议按以下顺序展示：

1. 任务触发区
2. 最新批次总览
3. 步骤状态表
4. 任务队列表
5. 分片明细
6. 后台执行器运维区
7. 错误和恢复动作区

## 6.6 当前实现

当前已具备：

1. 主流程任务触发
2. 单步任务触发
3. 批次与步骤查看
4. 任务队列与分片查看
5. 失败重试与恢复

## 6.7 当前短板

仍缺：

1. 更面向业务的错误分类
2. 更清晰的恢复建议
3. 更明显的“今天是否可进入下一步”提示

## 6.8 页面完成标准

一个合格的任务中心，不是把所有命令挂上去，而是用户能在几分钟内回答：

1. 跑没跑完
2. 卡在哪
3. 怎么补救

---

## 7. 页面三：执行回写

## 7.1 页面定位

执行回写是：

**真实成交进入系统的对账入口。**

## 7.2 页面回答的问题

用户进入这一页，应该能回答：

1. 哪些信号已经真实执行
2. 哪些还没执行
3. 成交录进去后，持仓有没有同步
4. 是否有 missed 的执行纪律问题

## 7.3 页面输入

主要来自：

1. trade_signal
2. execution_record
3. position
4. signal_execution_feedback

## 7.4 页面输出

用户离开页面前，至少应形成：

1. 所有已成交交易已回写
2. 持仓状态正确
3. 执行反馈无明显漏记

## 7.5 推荐模块结构

建议按以下顺序展示：

1. 手工回写表单
2. CSV 导入区
3. 今日信号区
4. 成交记录区
5. 执行反馈区
6. 持仓同步结果区

## 7.6 当前实现

当前已具备：

1. 手工录入
2. CSV 批量导入
3. 成交记录查询
4. 信号和反馈联动查询

## 7.7 当前短板

仍缺：

1. 部分成交表达
2. 多次成交归并表达
3. 信号和成交关联链可视化

## 7.8 页面完成标准

一个合格的执行回写页，不是能录数据，而是能让用户确认：

1. 实盘动作已经进入系统
2. 系统事实和真实交易一致

---

## 8. 页面四：影子对比

## 8.1 页面定位

影子对比是：

**策略灰度评审台，不是数字展示页。**

## 8.2 页面回答的问题

用户进入这一页，应该能回答：

1. 候选策略是否持续优于当前策略
2. 它优在哪里
3. 优势是否稳定
4. 是否应该 promote / keep / observe

## 8.3 页面输入

主要来自：

1. 基线策略回测结果
2. 候选策略回测结果
3. 月度聚合比较
4. canary 推荐结果

## 8.4 页面输出

用户离开页面前，至少应形成：

1. 当前基线是否继续保留
2. 候选是否进入观察或替换流程

## 8.5 推荐模块结构

建议按以下顺序展示：

1. 策略选择区
2. 月度对比总览
3. 关键指标对比
4. canary 推荐卡
5. 治理结论和建议动作区

## 8.6 当前实现

当前已具备：

1. 基线 / 候选选择
2. 月度对比结果
3. canary 推荐

## 8.7 当前短板

仍缺：

1. 更细粒度的适用范围解释
2. 图形化证据层
3. 更标准的策略切换 SOP 提示

## 8.8 页面完成标准

一个合格的影子对比页，不是能展示候选更强，而是能支撑：

1. 保留
2. 观察
3. 替换

这三类治理动作。

## 9. 页面之间的正式流转关系

## 9.1 任务中心 -> 量化看板

含义：

1. 先跑出结果
2. 再看结果

## 9.2 量化看板 -> 执行回写

含义：

1. 先形成执行清单
2. 次日执行后再回写

## 9.3 量化看板 / 执行回写 -> 复盘页面

含义：

1. 当日异常、典型交易、风险事件需要进一步解释

## 9.4 复盘页面 -> 影子对比

含义：

1. 日常复盘发现策略问题后，进入治理比较

---

## 10. 当前实现与目标标准

## 10.1 当前实现

当前四个页面已经覆盖了量化运营主线上的关键工作台：

1. 跑流程
2. 看结果
3. 回写成交
4. 看策略治理

这说明系统页面架构方向是对的。

## 10.2 当前缺口

当前最大问题不是没有页面，而是：

1. 页面说明不足
2. 页面之间缺少明确流转
3. 每个页面的“完成标准”没有被定义

## 10.3 目标标准

理想状态下，任意一个新用户都应能根据页面说明完成：

1. 盘后跑批
2. 看信号与风险
3. 回写成交
4. 做复盘与策略治理

而不需要先理解整套代码。

---

## 11. 一句话总结

当前量化系统的页面，不应再被理解为四个孤立菜单，而应正式定义成一条：

**任务运行 -> 结果查看 -> 成交回写 -> 复盘治理**

的低频量化操作链。
