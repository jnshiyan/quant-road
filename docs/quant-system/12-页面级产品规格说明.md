# 页面级产品规格说明

> 版本：V1.0  
> 日期：2026-05-06  
> 范围：基于 `08-复盘体系与图表需求说明` 与 `09-页面体系与操作流程说明`，将量化分析模块正式落成页面级产品规格，按页面输出组件清单、字段清单、接口清单和交互流程。

---

## 1. 规格目标

本规格不再停留在“页面怎么理解”，而是正式回答以下问题：

1. 当前量化分析模块由哪些正式页面组成
2. 每个页面分别承担什么业务职责
3. 每个页面应包含哪些组件
4. 每个组件消费和输出哪些字段
5. 页面依赖哪些现有接口
6. 为了补齐产品闭环，还需要新增哪些接口
7. 页面之间如何形成一条连续操作链

本规格覆盖 5 个页面：

1. `任务中心`
2. `量化看板`
3. `执行回写`
4. `复盘页面`
5. `影子对比`

---

## 2. 页面总图

这 5 个页面不是 5 个平行菜单，而是一条低频量化操作链：

1. `任务中心`
   - 跑出盘后生产结果，确认是否可进入下一步
2. `量化看板`
   - 查看当日市场、信号、持仓、风险和治理提示
3. `执行回写`
   - 将真实成交写回系统，完成实盘对账
4. `复盘页面`
   - 解释为什么买、为什么卖、为什么异常、为什么治理
5. `影子对比`
   - 对候选策略和基线策略做升级评审

按使用频率可分成两类：

1. 运营页
   - `任务中心`
   - `量化看板`
   - `执行回写`
2. 治理页
   - `复盘页面`
   - `影子对比`

---

## 3. 全局口径定义

### 3.1 核心对象

本规格统一围绕 6 个核心对象展开：

1. `策略`
   - 产生日志、信号、回测结果、治理结论的主体
2. `信号`
   - 某交易日、某标的、某策略产生的 `BUY / SELL / HOLD / BLOCKED` 结果
3. `执行记录`
   - 真实成交及其与信号的关联关系
4. `持仓快照`
   - 当前或历史某时点的仓位、成本、收益和风险状态
5. `复盘案例`
   - 一次正式复盘对象，可以是单标的、单策略、组合或治理对象
6. `治理比较`
   - 基线策略与候选策略在一段周期内的证据对比

### 3.2 全局状态枚举

为避免不同页面口径不一致，本规格统一以下状态：

1. `market_status`
   - `bull / volatile / bear / panic`
2. `signal_type`
   - `BUY / SELL / HOLD / BLOCKED`
3. `execution_status`
   - `pending / partial / executed / missed / cancelled / unmatched`
4. `exit_reason`
   - `strategy_exit / stop_loss_exit / manual_execution_exit / pending_unresolved`
5. `review_conclusion`
   - `healthy / observe / warning / invalid`
6. `governance_action`
   - `keep / reduce_weight / observe / replace / disable`
7. `job_status`
   - `queued / pending / running / success / partial_failed / failed / cancelled`

### 3.3 全局筛选维度

后续各页面应优先复用以下统一筛选条件：

1. `strategyId`
2. `stockCode / assetCode`
3. `signalDate / tradeDate / reviewDate`
4. `dateRange`
5. `marketStatus`
6. `executionStatus`
7. `reviewLevel`
   - `trade / strategy / portfolio / governance`

### 3.4 跨页动作链

本系统正式定义 4 条跨页动作链：

1. `任务中心 -> 量化看板`
   - 盘后流程成功，或达到业务上可查看状态
2. `量化看板 -> 执行回写`
   - 看板形成次日执行清单后，次日将真实成交回写
3. `量化看板 / 执行回写 -> 复盘页面`
   - 出现异常信号、漏执行、止损、连续亏损、风险预警时进入复盘
4. `复盘页面 -> 影子对比`
   - 复盘结论进入 `observe / replace / disable`，需要进一步治理时进入影子对比

### 3.5 页面完成判定

各页面不以“数据能展示”为完成标准，而以“用户能形成下一步动作”为完成标准：

1. 运营页完成标准
   - 用户能形成下一步操作
2. 治理页完成标准
   - 用户能形成明确治理结论
3. 所有关键结论都必须能落到审计事件或建议动作

### 4.5 交互流程

#### 4.5.1 日常盘后主流程

1. 用户进入页面
2. 系统默认加载策略列表、任务队列、最新批次
3. 用户配置执行范围并点击 `执行任务`
4. 页面返回执行回执或任务回执
5. 如返回 `jobId` 或 `executionId`，页面自动切到当前任务状态卡并轮询
6. 当任务完成且业务就绪状态为 `ready` 时，页面出现“进入量化看板”主按钮

#### 4.5.2 异常补救流程

1. 用户在批次列表中看到 `FAILED`
2. 点击失败批次查看步骤明细
3. 页面高亮失败步骤，并显示错误分类与建议补救动作
4. 用户执行重试失败分片、取消任务或重新触发任务
5. 补救成功后，页面刷新批次状态，并更新“是否可进入下一步”

#### 4.5.3 治理任务流程

1. 用户选择基线策略、候选策略、统计月份
2. 触发 `shadowCompare` 或 `canaryEvaluate`
3. 页面展示回执，并将结果写入后续 `影子对比` 与 `量化看板`
4. 成功后提供跳转入口到 `影子对比`

### 4.6 页面完成标准

用户必须能在 3 分钟内确认：

1. 今天主流程是否完成
2. 卡在哪一步
3. 应该补救什么
4. 能不能进入 `量化看板`

### 5.5 交互流程

#### 5.5.1 盘后查看主流程

1. 用户进入看板
2. 系统按默认日期加载总览、市场状态、信号、持仓、反馈、canary、日志
3. 用户先看顶部总览卡和市场状态
4. 再看当日信号表，形成次日候选执行清单
5. 再看持仓表和执行反馈，确认是否存在异常仓位或漏执行
6. 最后看 canary 与策略日志，判断是否需要进入治理或深度复盘

#### 5.5.2 信号到执行流程

1. 用户在信号表看到 `BUY / SELL`
2. 点开信号解释抽屉或跳转详情
3. 若信号待执行，则加入“次日执行清单”
4. 用户点击“去执行回写”时，带上 `signalId / stockCode / strategyId / signalType` 跳转

#### 5.5.3 异常到复盘流程

1. 用户在看板发现 `missed`、止损预警、连续亏损策略或 canary 异常建议
2. 点击“进入复盘”
3. 页面带上复盘对象类型与筛选参数跳转到 `复盘页面`
4. 复盘结论回写后，看板应能展示“已复盘 / 待处理”状态

#### 5.5.4 治理触发流程

1. 用户从日志或 canary 发现策略进入 `observe / warning / invalid`
2. 点击“进入影子对比”
3. 带上 `baselineStrategyId / candidateStrategyId / months` 跳转
4. 在影子对比形成最终治理建议

### 5.6 页面完成标准

用户离开页面时至少形成 3 个结果：

1. 明天要执行什么
2. 现在要关注什么风险
3. 哪些对象需要进入复盘或治理

### 6.5 交互流程

2. 若带 `signalId` 进入，系统自动预填 `stockCode / side / strategyId / suggestPrice / signalDate`
3. 用户录入真实 `quantity / price / tradeDate / fees`
4. 提交后系统执行成交落库、自动匹配信号、更新信号执行状态、触发持仓同步、刷新执行反馈

#### 6.5.2 批量导入流程

1. 用户选择 CSV 上传或路径导入
2. 系统先做文件格式校验
3. 执行预校验并返回成功行、失败行、重复记录、未匹配信号
4. 用户确认导入
5. 系统落库并刷新执行记录、信号状态、反馈状态、持仓同步结果

#### 6.5.3 异常对账流程

1. 用户在反馈区看到 `pending / missed / partial`
2. 点击记录查看关联信号与成交
3. 若无成交，提示补录成交
4. 若部分成交，提示继续录入剩余数量
5. 若成交存在但未匹配，提示人工确认匹配
6. 用户处理后，反馈状态更新

#### 6.5.4 回写完成判定流程

1. 页面顶部显示今日待回写摘要
2. 当所有应执行信号都被处理为 `executed / missed / cancelled` 之一时，页面显示“今日回写完成”
3. 若仍有 `pending / unmatched / partial`，页面不允许标记完成

### 6.6 页面完成标准

用户必须能确认：

1. 哪些信号已经真实执行

4. 系统记录和真实交易是否一致

## 7. 页面四：复盘页面

### 7.1 页面定位

`复盘页面` 是本轮规格里新增定义的核心页。  
它的作用不是“多几张图”，而是把信号、执行、持仓、风险、治理串成一条：

1. 能解释
2. 能追责
3. 能形成下一步动作

### 7.2 复盘层级

复盘页统一支持 4 类复盘对象：

1. `单标的交易复盘`
2. `单策略行为复盘`
3. `组合与资金复盘`
4. `治理结果复盘`

### 7.3 组件清单

### 7.6 交互流程

#### 7.6.1 单标的交易复盘流程

1. 用户从看板信号、持仓或执行反馈进入复盘页
2. 页面默认落在 `trade` 层级，并带入 `stockCode / strategyId / signalId`
3. 顶部先显示复盘结论卡
4. 中部依次展示 `K 线 + 买卖点图`、`持仓区间图`、`事件时间线`
5. 用户判断信号是否符合规则、执行是否偏离、卖出是策略退出还是风控退出
6. 输出结论和建议动作

#### 7.6.2 单策略行为复盘流程

1. 用户从看板日志或 canary 异常进入
2. 页面切到 `strategy` 层级
3. 先看净值 + 基准 + 回撤图
4. 再看市场状态叠加图
5. 再看治理证据图和规则解释区
6. 形成策略行为结论

#### 7.6.3 组合与资金复盘流程

1. 用户选择 `portfolio` 层级
2. 页面展示仓位变化、预算分配、风险集中度
3. 用户判断总仓、单仓、行业集中度是否过高
4. 输出预算调整建议

#### 7.6.4 治理结果复盘流程

1. 用户从 `影子对比` 或 `canary` 进入
2. 页面切到 `governance` 层级
3. 展示治理证据图、时间线、结论卡
4. 用户判断是短期波动还是结构性失效
5. 输出 `keep / observe / replace / disable`

### 7.7 页面完成标准

用户必须能顺着统一判断链回答 5 个问题：

1. 信号是否符合策略规则
2. 信号是否通过执行约束
3. 持仓结果是否符合预期
4. 统计结果和图形行为是否一致
5. 是否需要形成治理动作

## 10. 统一交互、SOP 与状态流转

### 10.1 日常 SOP

1. 盘后先进入 `任务中心`
   - 完成条件：主流程成功，或系统给出可进入看板结论
2. 再进入 `量化看板`
   - 完成条件：形成次日执行清单、风险关注清单、待复盘清单
3. 次日进入 `执行回写`
   - 完成条件：真实成交回写完成、持仓同步成功、反馈状态清晰
4. 周 / 月进入 `复盘页面`
   - 完成条件：形成复盘结论和建议动作
5. 如涉及策略替换，再进入 `影子对比`

## 11. 建设优先级建议

结合当前实现状态，页面级规格建议按以下顺序落地：

1. 第一阶段：补最短闭环
   - `量化看板` 的建议动作区
   - `执行回写` 的对账摘要与持仓同步结果
   - `任务中心` 的就绪判断与错误分类
2. 第二阶段：落成复盘页
   - 先做 `复盘结论卡`
   - 再做 `K 线 + 买卖点图`
   - 再做 `净值 + 回撤图`
   - 最后补事件时间线和治理证据图
3. 第三阶段：治理升级
   - `影子对比` 的图形化证据层
   - 人工治理决策回写
   - 看板 / 复盘 / 影子对比三页联动

## 12. 一句话总结

本规格将 `08 + 09` 正式落成一套页面级产品定义：不再把量化分析理解成几个孤立页面，而是定义成一条：

**任务运行 -> 结果查看 -> 成交回写 -> 复盘解释 -> 策略治理**

的低频量化操作链，并为每个页面明确了组件、字段、接口和交互流程。
