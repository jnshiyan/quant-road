## 1. 这章要回答什么问题

如果不先把业务背景讲清楚，后面的看板、影子对比、任务中心、执行回写都会看起来像四个分散页面，而不是一套投资系统。

这一章要回答的不是“技术怎么实现”，而是：

1. 为什么要做这套系统
2. 它服务哪类投资者和哪类团队
3. 它到底解决的是“选股问题”还是“执行纪律问题”
4. 它为什么优先做盘后低频，而不是盘中自动交易
5. 页面上的几个模块，在真实投资流程里分别扮演什么角色

## 2. 原始业务问题是什么

这套系统要解决的，不是“如何找到一条神奇圣杯策略”，而是低频量化投资里最常见的五个实际问题。

### 2.1 问题一：人有判断，但没有稳定执行

很多投资者并不是完全没有方法，而是：

1. 知道应该顺势
2. 知道不该满仓
3. 知道亏损到一定程度该止损
4. 知道熊市不该硬扛

但真正执行时，常常会出现：

1. 盘中情绪化追买
2. 亏损后拖延卖出
3. 同一策略今天信、明天不信
4. 刚亏几次就临时改参数

所以这套系统首先解决的是：

**把投资纪律固化成可重复执行的规则。**

### 2.2 问题二：回测、信号、执行、复盘彼此脱节

很多量化系统的问题不是不会算，而是：

1. 回测报告是一套
2. 当日信号是一套
3. 实际成交又是一套
4. 复盘时无法知道这笔交易为什么发生

结果就是：

1. 纸面收益很好看
2. 实际执行很混乱
3. 出现问题时无法追责到具体环节

所以这套系统第二个核心目标是：

**让回测、信号、成交、持仓、反馈、复盘形成同一条闭环。**

### 2.3 问题三：上班族没有时间做盘中盯盘

这套系统默认服务的，不是职业日内交易员，而是：

1. 白天工作较忙
2. 只能盘后集中处理
3. 希望把交易压缩成少量高质量动作
4. 愿意用规则替代临场拍脑袋

这决定了系统的业务节奏必须是：

1. 盘后算
2. 次日执行
3. 周末复盘
4. 月度做策略治理

### 2.4 问题四：系统容易被“高收益想象”带偏

量化项目很容易一开始就被拉向：

1. 高频
2. 自动下单
3. 多因子
4. AI 黑盒
5. 参数寻优

但对于当前项目的真实阶段，这些东西反而容易造成：

1. 规则复杂度暴涨
2. 解释能力下降
3. 回测和实盘偏差变大
4. 系统维护成本失控

所以本系统的根本取舍是：

**先把低频、可解释、可执行、可复盘这四件事做扎实。**

### 2.5 问题五：策略不是唯一重点，边界和治理更重要

原始需求里最重要的一点，其实不是“再多加几条策略”，而是：

1. 标的边界要清楚
2. 回测范围要清楚
3. 风控优先级要清楚
4. 策略什么时候可用、什么时候该停要清楚

也就是说，这套系统不是以“预测”为中心，而是以“规则治理”为中心。

## 3. 这套系统服务谁

## 3.1 核心用户画像

当前最匹配的用户不是机构高频团队，而是以下三类。

### 第一类：个人低频投资者

特征：

1. 资金规模有限但希望长期稳定积累
2. 白天没有条件持续盯盘
3. 更重视回撤控制和执行纪律
4. 接受“少交易、慢迭代、重复盘”

### 第二类：小型量化运营团队

特征：

1. 需要统一盘后流程
2. 需要任务可追踪
3. 需要策略日志和执行反馈
4. 需要有人研究、有人执行、有人复盘时还能对齐口径

### 第三类：懂开发但不想靠主观炒股的人

特征：

1. 能理解系统和规则
2. 希望把投资流程工程化
3. 不想把交易建立在临场感觉上
4. 更愿意接受“系统约束自己”

## 3.2 不适合的用户和场景

这套系统当前不适合：

1. 高频交易
2. 分钟级、秒级择时
3. 盘中自动下单
4. 期货、期权、融资融券等复杂交易
5. 追求短期暴利、频繁切换题材的玩法

原因不是这些场景没有价值，而是它们对：

1. 数据质量
2. 下单通道
3. 延迟控制
4. 风险隔离
5. 订单生命周期管理

都有更高要求，不属于当前系统的合理边界。

## 4. 投资框架先讲结论

这套系统采用的，是一套**盘后低频、纪律优先、分层决策、人工执行确认**的投资框架。

可以概括成四句话：

1. 先看市场环境，再看策略是否允许运行
2. 先定交易对象边界，再做回测和信号
3. 先过仓位和风险约束，再允许 BUY 落地
4. 先复盘和治理，再决定是否切策略或改参数

它不是“预测市场下一分钟怎么走”的系统，而是：

**在不追求高频反应速度的前提下，把选对象、跑策略、控风险、管执行、做复盘连成一条纪律化流程。**

## 5. 正式投资边界

这部分必须写死，否则系统越做越漂。

## 5.1 交易对象边界

当前正式交易对象只有两类：

1. A股个股
2. A股ETF

## 5.2 指数边界

指数不是当前系统的直接交易对象，而承担四种职责：

1. 市场状态判断
2. 估值观察
3. ETF 上游映射
4. 策略基准比较

这意味着：

1. 指数负责“看环境”
2. ETF 负责“做指数投资”
3. 个股负责“做主动选股”

## 5.3 时间粒度边界

当前系统以日线和周线复盘逻辑为主，不以分钟级策略为核心。

业务含义是：

1. 盘后数据更新
2. 盘后生成信号
3. 次日再执行
4. 周末看阶段表现

## 5.4 执行边界

当前系统输出：

1. 信号
2. 风险提示
3. 成交回写后的持仓同步
4. 执行反馈

但它不直接承担：

1. 券商自动下单
2. 盘中追单
3. 高频订单管理

所以它更准确的定位是：

**半自动量化运营系统，而不是全自动交易系统。**

## 6. 这套系统真正想建立的，不是“策略集合”，而是“投资作业框架”

这点非常关键。

从业务角度看，系统不是先问“今天买什么”，而是按下面顺序工作。

## 6.1 第一步：今天处在什么市场环境

先回答：

1. 当前是 bull、volatile、bear 还是 panic
2. 哪些策略今天允许跑
3. 总资金预算应偏进攻还是偏保守

这一步的意义是：

1. 不让策略脱离市场环境裸跑
2. 不让投资行为只由个别标的信号决定

## 6.2 第二步：今天在哪些对象里寻找机会

再回答：

1. 这次是看个股池，还是看 ETF 池
2. 回测或执行范围是否只看白名单
3. 是否存在黑名单、临时剔除项

这一步的意义是：

1. 不让“全市场默认值”掩盖真实投资边界
2. 让研究范围和执行范围尽量一致

## 6.3 第三步：今天策略在这些对象上会产生什么信号

这一步才进入回测和信号生成：

1. 加载历史数据
2. 计算策略信号
3. 输出最新 BUY / SELL
4. 记录收益、回撤、胜率、交易次数等指标

## 6.4 第四步：这些信号是不是可执行

BUY 信号不是算出来就能用，还要过：

1. 总仓位限制
2. 单标的仓位限制
3. 是否够买一手
4. 市场状态是否允许

这一步解决的是：

1. 理论信号和现实资金约束之间的落差

## 6.5 第五步：执行后如何回到系统里

真实交易完成后，还要把：

1. 成交价格
2. 成交数量
3. 手续费
4. 对应信号

回写到系统里，更新持仓和执行状态。

否则：

1. 风控会失真
2. 看板会失真
3. 复盘会失真

## 6.6 第六步：事后如何复盘和治理

最后不是只看当天盈亏，而是要回答：

1. 策略最近是否持续有效
2. 是否连续亏损过多
3. 是否持续跑输基准
4. 候选策略是否值得替换现有策略

这一步才是“量化系统真正长期有效”的关键。

## 7. 页面在业务里的真实定位

你最开始没看懂页面怎么用，根本原因不是页面本身复杂，而是业务定位没有写清楚。

下面用业务语言重新定义四个页面。

## 7.1 量化看板：盘后总览，不是交易终端

它解决的是：

1. 今天市场环境怎么样
2. 今天有哪些信号
3. 当前有哪些持仓
4. 最近策略表现如何

它不是用来盘中做点买卖决策的，而是盘后和值班时看的总览页。

业务角色类似：

1. 日报首页
2. 盘后运营首页

## 7.2 影子对比：策略换将评审台，不是研究玩具

它解决的是：

1. 候选策略有没有比基线更稳
2. 是该替换、继续观察，还是维持原策略

它的本质不是“看看数字哪个好看”，而是：

**给策略升级提供一个尽量克制、可审计的流程。**

## 7.3 任务中心：盘后生产控制台

它解决的是：

1. 今天的盘后流程有没有跑完
2. 是哪一步失败了
3. 失败后怎么重试
4. 长任务现在执行到哪了

这页更像：

1. 投资系统的生产调度台
2. 值班排障台

## 7.4 执行回写：实盘与系统对账入口

它解决的是：

1. 信号有没有真实执行
2. 成交怎么进系统
3. 持仓怎么同步
4. 哪些信号过期未执行

没有这页，系统就只是“会出建议”，不是“能管理执行闭环”。

## 8. 低频量化为什么优先选 ETF + 个股双轨，而不是只做一种

## 8.1 个股的意义

个股承担的是：

1. 主动收益来源
2. 更高 alpha 的可能性
3. 策略差异化空间

但问题是：

1. 波动更尖锐
2. 黑天鹅更强
3. 仓位控制要更严格

## 8.2 ETF 的意义

ETF 承担的是：

1. 指数观点的可交易承接
2. 风险更平滑的低频趋势交易对象
3. 比个股更容易规模化治理的一类资产

## 8.3 指数的意义

指数不直接交易，但它提供：

1. 市场环境解释
2. 估值和风格判断
3. ETF 分层管理的上游锚点

所以系统采用三层结构，不是为了概念好看，而是因为三类对象在投资里职责不同。

## 9. 为什么复盘优先，而不是实时盯盘优先

你之前点到“K线图没看到”，这个问题本质上不是图表缺失，而是系统没有先声明：

**这套图表是拿来复盘的，不是拿来盘中追价的。**

原因有三个。

### 9.1 原因一：当前系统主要服务盘后纪律

如果系统主节奏是：

1. 盘后跑批
2. 次日执行
3. 周末复盘

那么图表最先该支持的，就不是分时盯盘，而是：

1. 买卖点解释
2. 持仓区间解释
3. 净值和回撤解释
4. 市场状态切换解释

### 9.2 原因二：复盘能直接约束策略质量

很多策略数值上不差，但一看图就会发现：

1. 买点太迟
2. 卖点太乱
3. 震荡期反复打脸
4. 止损位置不合理

所以复盘图不是“好看组件”，而是质量检查工具。

### 9.3 原因三：当前阶段最该补的是解释能力

这套系统当前最大的缺口之一，不是公式不够，而是：

1. 为什么产生这个信号
2. 为什么今天没让买
3. 为什么这条策略被判 warning 或 invalid

图表最重要的价值，就是把这些逻辑讲明白。

## 10. 一套真正能指导低频量化投资的日常工作法

这一部分不讲技术，直接讲怎么用。

## 10.1 工作日盘后

标准动作应是：

1. 跑完 `full-daily`
2. 在量化看板确认市场状态、信号、持仓、风险提示
3. 在任务中心确认是否有失败步骤或异步任务异常
4. 形成次日执行清单

核心要求：

1. 盘后处理，而不是盘中乱改
2. 当天只看规则结果，不急着改策略

## 10.2 次日执行

标准动作应是：

1. 按系统前一日信号执行
2. 执行完把成交回写
3. 检查系统是否正确更新持仓和信号状态

核心要求：

1. 先执行再表达观点
2. 不因为开盘几分钟波动就全面推翻盘后结论

## 10.3 周末复盘

标准动作应是：

1. 看策略最近几周收益与回撤
2. 看连续亏损和失效触发情况
3. 看是否存在应该进入白名单或黑名单的标的
4. 看图复盘典型买卖点

核心要求：

1. 周末改研究和配置
2. 工作日做执行和核对

## 10.4 月度治理

标准动作应是：

1. 看影子对比
2. 看 canary 推荐
3. 判断现有策略是保留、观察还是替换
4. 输出月度结论而不是临时拍板

核心要求：

1. 参数调整频率低
2. 策略替换有证据链

## 11. 这套系统里最重要的业务逻辑，不是预测，而是约束

这句话需要单独强调。

从业务上看，本系统真正的价值排序更接近：

1. 先保证不乱买、不乱满仓、不乱拖延
2. 再保证信号、执行、持仓、反馈一致
3. 再保证策略可以比较、可以淘汰、可以替换
4. 最后才是逐步提高收益

所以这套系统不是一个“给你答案”的黑盒，而是一个“逼你按规则做事”的框架。

如果要用一句更准确的话来定义它，那就是：

**它是一套把投资纪律工程化的低频量化运营系统。**

## 12. 当前实现与目标标准

## 12.1 当前实现

从业务定位上，当前系统已经能看出一条真实主线：

1. 它已经是一套盘后低频量化系统
2. 它已经具备回测、信号、风控、执行回写、反馈、影子对比、任务治理这些基础骨架
3. 它已经明显不是“只做一个看板”
4. 它已经具备“投资作业框架”的雏形

但当前也必须诚实承认：

1. 页面业务解释还不够直观
2. ETF 层、池子层、回测范围层还没有完全产品化
3. 图表复盘能力还缺失
4. 用户第一次进入页面时，很难直接看懂每天该怎么用

## 12.2 目标标准

理想状态下，这一层应达到：

1. 用户能一眼看懂系统服务的投资场景
2. 用户能知道每天、每周、每月分别该做什么
3. 用户能分清指数、ETF、个股三层角色
4. 用户能把看板、任务中心、执行回写、影子对比放进同一条业务流程理解
5. 用户能理解这是“低频纪律系统”，而不是“自动赚钱机器”

---

## 13. 一句话总结

这套系统的出发点，不是帮用户做更刺激的交易，而是帮用户在 A股个股和 A股ETF 场景下，建立一套：

**盘后决策、次日执行、周末复盘、月度治理的纪律化低频量化投资框架。**
