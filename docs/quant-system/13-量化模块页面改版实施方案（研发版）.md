# 量化模块页面改版实施方案（研发版）

> 版本：V1.0  
> 日期：2026-05-06  
> 适用对象：前端、后端、测试、产品  
> 目标：基于现有量化模块，完成信息架构调整、首屏重构、少量聚合接口补齐，并按阶段落地。

---

## 1. 实施目标

本次实施不以“新增更多功能”为目标，而以“重组现有能力，让页面更符合业务操作链”为目标。

本次实施要解决的问题：

1. 当前菜单与页面层级未区分日常运营和策略治理
2. 多数页面首屏信息主次不清，表格和日志优先级过高
3. 用户难以在短时间内判断下一步动作
4. 任务异常、执行异常、治理建议缺少统一入口
5. 复盘分析缺少正式页面承载

---

## 2. 实施范围

本次涉及以下页面和模块：

1. 菜单结构调整
2. `量化看板` 首屏重构
3. `任务中心` 首屏重构
4. `执行回写` 首屏重构
5. `影子对比` 首屏优化
6. `复盘分析` V1 新增页面
7. 公共组件抽象
8. 少量后端聚合接口新增

---

## 3. 菜单与路由改造

## 3.1 目标结构

一级结构调整为：

1. `量化运营`
   - `量化看板`
   - `任务中心`
   - `执行回写`

2. `策略治理`
   - `复盘分析`
   - `影子对比`

默认首页：
`量化看板`

## 3.2 前端改造项

1. 更新菜单配置或菜单初始化 SQL
2. 调整菜单排序
3. 增加 `复盘分析` 路由
4. 设置量化模块默认 landing page 为 `量化看板`

## 3.3 数据改造项

如使用 `sys_menu` 初始化脚本，需要新增/调整：

1. `量化运营` 目录
2. `策略治理` 目录
3. `复盘分析` 菜单项
4. 原 `量化分析` 下已有子菜单的父级迁移

## 3.4 验收标准

1. 左侧菜单可明显区分运营页和治理页
2. `量化看板` 成为默认进入页
3. 原页面功能仍可访问
4. 菜单权限不因迁移失效

## 4. 页面实施方案

## 4.1 量化看板

### 4.1.1 页面目标

让用户在 3 分钟内完成以下判断：

1. 今天市场环境如何
2. 明天是否需要执行 BUY / SELL
3. 当前是否存在风险或异常
4. 是否需要进入任务中心或执行回写

### 4.1.2 首屏布局

1. 顶部状态卡
   - 市场状态
   - 待执行信号
   - 风险预警
   - 最新任务状态

2. 主区左列
   - 今日信号清单

3. 主区右列
   - 待处理事项

4. 下方摘要区
   - 当前持仓摘要

5. 下沉区
   - 市场估值概览
   - 最近策略日志

### 4.1.3 前端组件拆分

1. `QuantDashboardHeader`
2. `QuantDashboardSummaryCards`
3. `QuantSignalTable`
4. `QuantTodoPanel`
5. `QuantPositionSummary`
6. `QuantMarketSnapshot`
7. `QuantStrategyLogList`

### 4.1.4 接口映射

复用接口：

1. `GET /quant/dashboard/summary`
2. `GET /quant/data/marketStatus`
3. `GET /quant/data/indexValuations`
4. `GET /quant/data/signals`
5. `GET /quant/data/positions`
6. `GET /quant/data/strategyLogs`
7. `GET /quant/data/executionFeedbackSummary`
8. `GET /quant/data/jobBatches`
9. `GET /quant/data/strategySwitchAudits`

建议新增：

1. `GET /quant/data/todoCenter`

### 4.1.5 联调重点

1. `signals` 是否支持按日期、状态筛出“今日重点信号”
2. `dashboard/summary` 是否足够支撑顶部 4 卡
3. `todoCenter` 若未实现，前端需先聚合执行异常、任务异常、治理提醒
4. 信号行跳转动作需统一路由参数格式

### 4.1.6 验收标准

1. 首屏必须完整展示状态卡、信号主表、待处理事项
2. 日志模块不得占首屏主视觉
3. 用户可从信号直接跳到 `执行回写` 或 `复盘分析`
4. 用户可从异常项直接跳到 `任务中心`

---

## 4.2 任务中心

### 4.2.1 页面目标

让用户在 1 分钟内完成以下判断：

1. 今日主流程有没有完成
2. 当前流程卡在哪一步
3. 是否已满足进入看板条件
4. 失败项如何恢复

### 4.2.2 首屏布局

1. 主流程状态区
2. 标准任务操作区
3. 当前批次步骤流
4. 异常与恢复区
5. 异步任务摘要
6. 详情抽屉

### 4.2.3 前端组件拆分

1. `QuantJobCenterHeader`
2. `QuantPipelineHero`
3. `QuantJobActionPanel`
4. `QuantBatchStepTimeline`
5. `QuantRecoveryPanel`
6. `QuantAsyncJobSummary`
7. `QuantJobDetailDrawer`

### 4.2.4 接口映射

复用接口：

1. 所有 `POST /quant/jobs/*`
2. `GET /quant/jobs/status/{jobId}`
3. `POST /quant/jobs/cancel/{jobId}`
4. `POST /quant/jobs/retryFailedShards/{jobId}`
5. `GET /quant/data/jobBatches`
6. `GET /quant/data/jobSteps`
7. `GET /quant/data/asyncJobs`
8. `GET /quant/data/asyncJobShards`
9. `GET /quant/data/asyncJobResults`
10. `GET /quant/data/strategies`

建议新增：

1. `GET /quant/data/jobReadiness`
2. `GET /quant/data/jobErrorCategories`
3. `POST /quant/jobs/recoverBatch/{batchId}`
4. `GET /quant/data/jobSopHints`

### 4.2.5 联调重点

1. `jobBatches + jobSteps` 是否可唯一定位“今日主流程”
2. `jobReadiness` 需要给出明确状态，例如：
   - `not_started`
   - `running`
   - `partial_ready`
   - `ready`
   - `failed`
3. `jobErrorCategories` 需要给前端返回可展示的业务分类，而不是原始异常堆栈
4. 各任务按钮需要统一 loading、回执、轮询逻辑

### 4.2.6 验收标准

1. 用户进入页面即可判断当天流程状态
2. 用户可快速找到失败步骤和失败任务
3. 用户可直接执行重试或恢复动作
4. 首屏不以多张大表为主要结构

## 4.3 执行回写

### 4.3.1 页面目标

让用户快速完成“信号 -> 成交”的闭环录入。

### 4.3.2 首屏布局

1. 顶部状态卡
2. 待回写信号主表
3. 手工录入 / CSV 导入区
4. 执行异常提醒
5. 最近成交摘要

### 4.3.3 前端组件拆分

1. `QuantExecutionHeader`
2. `QuantExecutionSummaryCards`
3. `QuantPendingExecutionTable`
4. `QuantExecutionEntryPanel`
5. `QuantExecutionExceptionPanel`
6. `QuantRecentExecutionTable`
7. `QuantExecutionHistoryDrawer`

### 4.3.4 接口映射

复用接口：

1. `GET /quant/data/signals`
2. `GET /quant/data/executionRecords`
3. `GET /quant/data/executionFeedbackSummary`
4. `GET /quant/data/executionFeedbackDetails`
5. `POST /quant/jobs/recordExecution`
6. `POST /quant/jobs/importExecutions`
7. `POST /quant/jobs/importExecutionsUpload`

建议新增：

1. `GET /quant/data/pendingExecutions`

### 4.3.5 联调重点

1. 若没有 `pendingExecutions`，前端需要从 `signals` 中自行筛出待回写数据
2. “带入录入”动作需要约定字段映射：
   - 标的
   - side
   - quantity
   - signalId
   - tradeDate
3. 导入成功后需统一刷新：
   - 状态卡
   - 待回写表
   - 异常提醒
   - 最近记录

### 4.3.6 验收标准

1. 首屏先展示待回写信号
2. 用户可从待回写信号一键带入录入
3. 手工录入与 CSV 导入都可在同页完成
4. 用户可快速定位 missed / partial / unmatched

## 4.5 复盘分析 V1

### 4.5.1 页面目标

把分散在信号、执行、日志、风险中的素材收敛为“结论 + 证据”的复盘页面。

### 4.5.2 首屏布局

1. 复盘对象栏
2. 复盘结论卡
3. 信号与行情回看
4. 持仓与收益回看
5. 风险与执行偏差
6. 原始日志附录

### 4.5.3 前端组件拆分

1. `QuantReviewHeader`
2. `QuantReviewFilterBar`
3. `QuantReviewConclusionCard`
4. `QuantReviewSignalPanel`
5. `QuantReviewPnLPanel`
6. `QuantReviewRiskPanel`
7. `QuantReviewAppendixCollapse`

### 4.5.4 接口映射

复用接口：

1. `GET /quant/data/signals`
2. `GET /quant/data/positions`
3. `GET /quant/data/strategyLogs`
4. `GET /quant/data/executionRecords`
5. `GET /quant/data/executionFeedbackDetails`
6. `GET /quant/data/marketStatus`

建议新增：

1. `GET /quant/data/reviewCases`
2. `GET /quant/data/reviewCaseDetail/{id}`

### 4.5.5 联调重点

1. V1 不建议前端自行从多个原始接口拼“复盘结论”
2. `reviewCaseDetail` 需由后端直接返回：
   - 结论
   - 证据列表
   - 建议动作
   - 附加原始数据引用
3. 复盘对象至少需要覆盖：
   - 异常执行
   - 风险预警
   - 止损/退出
   - 连续亏损观察对象

### 4.5.6 验收标准

1. 页面首屏必须先出现复盘结论卡
2. 用户可根据复盘对象直接进入详情
3. 页面能输出建议动作，而不只是展示日志
4. 原始日志和附加信息下沉到底部

## 9. 风险与注意事项

1. 当前若前端 heavily 依赖原始接口字段，聚合逻辑需避免与旧逻辑冲突
2. 菜单迁移要注意权限继承和已有角色绑定
3. `复盘分析` 若无后端结论接口支持，前端 V1 很容易退化成“又一个明细页”
4. `任务中心` 若不补 `jobReadiness`，产品化效果会明显打折
5. `todoCenter` 若完全靠前端聚合，维护成本较高

---

## 10. 完成定义

当以下条件同时满足时，可认为本轮实施完成：

1. 菜单已完成运营/治理分组
2. 默认首页切换为 `量化看板`
3. `量化看板`、`任务中心`、`执行回写` 首屏逻辑完成重构
4. `影子对比` 完成结论前置
5. `复盘分析` V1 可正常使用
6. 关键跨页链路可走通
7. 至少 5 个新增/重组核心模块通过联调与回归验证
