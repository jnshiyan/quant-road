# 量化分析模块目标产品蓝图

> 版本：V1.0
> 日期：2026-05-12
> 主题：基于 `docs/quant-system/*` 已确认的业务背景、投资框架、页面规格与实施约束，重新定义量化分析模块的目标产品形态，作为后续信息架构调整、页面重构与交互收敛的统一依据。

---

## 1. 文档目标

本文档不再讨论局部页面优化，而是明确回答以下问题：

1. 从低频量化投资者视角，这套系统最终应该长什么样
2. 哪条业务主线是产品唯一主线
3. 量化分析模块应保留哪些正式页面
4. 每个页面只负责什么，不负责什么
5. 用户每天、每周、每月如何使用这套系统
6. 哪些全局约束必须写死，避免页面重新退化为说明页、PPT 页或大表拼装页

本文档是目标态设计，不等同于当前实现。

---

## 2. 产品定位

### 2.1 正式定位

本系统不是高频交易终端，不是自动下单平台，也不是后台功能集合，而是：

**一套把投资纪律工程化的低频量化运营系统。**

### 2.2 适用场景

本系统服务的核心场景是：

1. 盘后决策
2. 次日执行
3. 周度复盘
4. 月度治理

它默认服务的不是盘中盯盘交易员，而是：

1. 个人低频投资者
2. 小型量化运营团队
3. 懂开发但希望用规则约束投资行为的人

### 2.3 核心价值

本系统最重要的价值不是预测，而是约束与闭环：

1. 先保证不乱买、不乱满仓、不乱拖延
2. 再保证信号、成交、持仓、反馈一致
3. 再保证策略可比较、可观察、可替换
4. 最后才是逐步提升收益

---

## 3. 正式业务边界

### 3.1 交易对象边界

当前正式交易对象只有两类：

1. A股个股
2. A股ETF

### 3.2 指数边界

指数不是当前直接交易对象，只承担以下职责：

1. 市场环境判断
2. 估值观察
3. ETF 上游映射
4. 策略基准比较

正式口径必须统一为：

1. 指数负责看环境
2. ETF 负责承接指数投资
3. 个股负责主动选股

### 3.3 时间节奏边界

系统默认按低频节奏运行：

1. 盘后更新数据
2. 盘后生成信号
3. 次日执行
4. 周度复盘
5. 月度治理

### 3.4 执行边界

系统输出的是：

1. 信号
2. 风险提示
3. 成交回写后的持仓同步
4. 执行反馈
5. 治理结论

系统不直接承担：

1. 券商自动下单
2. 盘中追单
3. 高频订单生命周期管理

---

## 4. 唯一主线

本系统的唯一主线不是“看板 / 调度 / 运维 / 复盘”并列，而是以下投资作业链：

1. 先看市场环境
2. 再看今日范围
3. 再看策略信号
4. 再看执行约束
5. 再做真实执行与对账
6. 再做复盘解释
7. 最后进入策略治理

对应的正式产品链为：

**盘后决策 -> 调度中心 -> 执行对账 -> 复盘分析 -> 策略治理**

说明：

1. `盘后决策` 是默认首页
2. `调度中心` 是流程控制平面
3. `执行对账` 是实盘闭环入口
4. `复盘分析` 是解释层
5. `策略治理` 是月度和阶段性决策层

## 5. 顶层信息架构

### 5.1 正式页面

量化分析模块正式收敛为 5 个页面：

1. `盘后决策`
2. `调度中心`
3. `执行对账`
4. `复盘分析`
5. `策略治理`

### 5.2 页面分组

按用户任务分组，而不是按系统实现分组：

1. 日常运营页
   - `盘后决策`
   - `调度中心`
   - `执行对账`
2. 解释与治理页
   - `复盘分析`
   - `策略治理`

### 5.3 页面映射建议

对现有页面建议做如下映射：

1. `量化看板` -> `盘后决策`
2. `任务中心` -> `调度中心`
3. `执行回写` -> `执行对账`
4. `复盘分析` -> 保留为正式页面
5. `影子对比` -> `策略治理`

### 5.4 下沉页面与技术页处理建议

以下现有能力不应再作为主导航一层页面暴露：

1. 手工调度详情
2. 自动调度详情
3. 任务详情
4. 运维工具页
5. 原始 shadow 对比明细页
6. 原始 jobs 页面
7. 原始 dispatch 页面
8. 原始 backtest 工具页

它们应当：

1. 并入正式主页面
2. 作为详情抽屉或二级页存在
3. 或转为配置页、工具页，不再承担用户主链职责

## 6. 页面一：盘后决策

### 6.1 页面定位

`盘后决策` 是默认首页，是盘后结果总览和次日执行清单生成页，不是盘中交易终端，也不是后台总控台。

### 6.2 页面唯一目标

用户进入本页后，应在 30 秒到 3 分钟内完成 4 个判断：

1. 今天环境是否支持动作
2. 今天看的范围是什么
3. 今天有哪些应关注的信号
4. 今天有哪些阻断项和待处理事项

### 6.3 首屏结构

首屏只允许 4 个核心区块：

1. `今日状态`
2. `今日信号`
3. `待处理事项`
4. `当前持仓`

### 6.4 首屏最小信息集

#### 今日状态

只显示：

1. 交易日
2. 市场状态
3. 当前观察范围
4. 最新调度状态
5. 今日待处理数

#### 今日信号

作为首屏主表，必须优先显示：

1. 标的代码与名称
2. 资产类型
3. 信号类型
4. 信号日期
5. 信号原因
6. 风险阻断原因
7. 执行状态
8. 下一步动作

#### 待处理事项

只显示会影响下一步动作的事项：

1. 调度失败
2. 漏执行
3. 风险预警
4. 待回写成交
5. 待复盘对象
6. 待治理策略

#### 当前持仓

只显示摘要，不展示大表：

1. 持仓数
2. 总仓位
3. 风险预警数
4. 高风险持仓数
5. 需要优先关注的少量持仓

### 6.5 不应放在首屏的内容

以下内容必须下沉：

1. 长任务明细
2. 分片明细
3. 大段策略日志
4. 详细估值表
5. 治理报表
6. 长篇说明文
7. 复盘证据图全量展开

### 6.6 页面输出

用户离开本页时，必须形成两份结果：

1. 明日执行清单
2. 今日待处理清单

---

## 7. 页面二：调度中心

### 7.1 页面定位

`调度中心` 是流程页和控制平面，不是交易页。

### 7.2 页面唯一目标

用户进入本页后，应快速判断：

1. 今天盘后流程有没有跑
2. 当前跑到哪一步
3. 当前跑的范围、时间范围、策略范围是什么
4. 如失败，应如何恢复

### 7.3 首屏结构

首屏只允许以下 5 个区块：

1. `今日调度状态`
2. `当前运行`
3. `最近结果`
4. `恢复动作`
5. `历史记录入口`

### 7.4 当前运行必须显示的内容

当存在运行中任务时，必须明确展示：

1. 当前任务名
2. 当前步骤名
3. 当前处理范围
4. 当前时间范围
5. 完成进度
6. 最新日志摘要
7. 下一步是什么

### 7.5 不应放在本页主视觉的内容

1. 今日买卖信号主表
2. 当前持仓主表
3. 策略治理建议
4. 成交录入功能

### 7.6 页面输出

用户离开本页时，必须能回答：

1. 今天是否已业务就绪
2. 是否可进入 `盘后决策`
3. 如果不可以，下一步补救动作是什么

---

## 8. 页面三：执行对账

### 8.1 页面定位

`执行对账` 是实盘成交与系统信号、持仓事实的闭环页，不是策略页。

### 8.2 页面唯一目标

用户进入本页后，应快速判断：

1. 哪些信号已经执行
2. 哪些没有执行
3. 哪些成交未匹配
4. 持仓是否同步正确

### 8.3 首屏结构

首屏只允许以下 5 个区块：

1. `今日对账状态`
2. `待处理信号`
3. `成交录入`
4. `异常对账`
5. `最近结果`

### 8.4 待处理信号主表最小字段集

1. 标的
2. 资产类型
3. BUY / SELL
4. 信号日期
5. 建议数量或应执行数量
6. 当前状态
7. 下一步动作

### 8.5 支持的异常类型

至少支持以下状态和异常：

1. 待执行
2. 部分执行
3. 已执行
4. 已错过
5. 未匹配
6. 持仓同步失败

### 8.6 页面输出

用户离开本页时，必须能确认：

1. 哪些信号已经完成对账
2. 哪些仍待处理
3. 系统持仓是否与真实交易一致

---

## 9. 页面四：复盘分析

### 9.1 页面定位

`复盘分析` 是解释页，不是操作总控页。

### 9.2 页面唯一目标

用户进入本页后，应沿同一条判断链完成解释：

1. 信号为什么产生
2. 信号为什么没执行或执行偏差
3. 持仓期间发生了什么
4. 图形行为和统计结果是否一致
5. 是否需要形成治理动作

### 9.3 首屏结构

首屏只允许以下 4 个区块：

1. `复盘结论`
2. `核心证据`
3. `规则解释`
4. `事件时间线`

### 9.4 复盘对象层级

本页统一支持 4 类对象：

1. 单标的交易复盘
2. 单策略行为复盘
3. 组合与资金复盘
4. 治理结果复盘

### 9.5 策略失效在本页的职责

本页不负责最终治理决策，但必须负责解释：

1. 为什么出现 `warning`
2. 为什么出现 `invalid`
3. 触发了哪些失效条件
4. 这些条件在图形和时间线上如何体现

### 9.6 页面输出

用户离开本页时，必须形成：

1. 复盘结论
2. 建议动作
3. 是否需要进入 `策略治理`

---

## 10. 页面五：策略治理

### 10.1 页面定位

`策略治理` 是正式决策页，不是研究报表页。

### 10.2 页面唯一目标

用户进入本页后，应快速判断：

1. 当前策略是否还应继续
2. 候选策略是否真的更优
3. 优势是否稳定
4. 最终治理动作是什么

### 10.3 首屏结构

首屏只允许以下 5 个区块：

1. `治理结论`
2. `对比摘要`
3. `核心证据`
4. `适用范围解释`
5. `治理动作`

### 10.4 策略健康状态正式表达

策略健康状态统一分为三档：

1. `健康`
2. `观察`
3. `失效`

### 10.5 策略失效在系统中的落位

策略是否失效必须分三层体现：

1. `盘后决策`
   - 做提醒
2. `复盘分析`
   - 做证据解释
3. `策略治理`
   - 做正式结论和治理动作

### 10.6 页面输出

用户离开本页时，必须形成正式治理决策：

1. 保留
2. 观察
3. 替换
4. 停用

## 11. 用户使用 SOP

### 11.1 日常节奏

#### 工作日盘后

1. 先进入 `盘后决策`
2. 如发现流程未完成或失败，进入 `调度中心`
3. 形成明日执行清单与今日待处理清单

#### 次日执行后

1. 进入 `执行对账`
2. 录入或导入成交
3. 确认信号、成交、持仓一致

#### 异常场景

1. 从 `盘后决策` 或 `执行对账` 进入 `复盘分析`
2. 对异常、止损、漏执行、策略异常做解释

### 11.2 周度节奏

每周主要完成：

1. 典型交易复盘
2. 持仓与风险检查
3. 观察策略筛选

### 11.3 月度节奏

每月主要完成：

1. 策略健康状态检查
2. 候选与基线对比
3. 正式治理动作确认

### 11.4 用户工作法总结

系统最终应服务以下节奏：

**日看决策与执行，周看行为与风险，月看策略与治理。**

## 12. 全局交互原则

### 12.1 页面必须自解释

所有页面必须通过以下元素完成自解释：

1. 页面结构
2. 模块命名
3. 字段命名
4. 状态标签
5. 按钮动作
6. 结果反馈

### 12.2 严禁说明型页面

以下做法在目标态中应明确禁止：

1. 大段陈述句
2. 长篇说明文
3. PPT 式大 banner、大 span、大标题占首屏
4. 用说明文弥补页面结构不清

### 12.3 先结论，后证据

所有页面遵守以下原则：

1. 首页先给动作清单
2. 复盘页先给结论
3. 治理页先给治理动作
4. 证据和原始数据下沉

### 12.4 先动作，后信息

页面首先要让用户知道下一步做什么，而不是先展示大量信息。

### 12.5 首屏只服务一个主判断

每个页面首屏只能回答一种主问题，不能同时承担多个主任务。

## 13. 全局硬约束

### 13.1 所有列表必须分页

这是全模块正式硬约束。

所有列表一律遵守：

1. 默认分页
2. 默认页大小保守
3. 必须有默认排序
4. 只展示支持当前决策的最小字段集
5. 每行都必须支持下一步动作

不允许：

1. 默认全量展开
2. 在首屏堆多张大表
3. 用无限下拉替代分页

### 13.2 首屏长度约束

所有正式页面都必须满足：

1. 首屏只允许一个主表
2. 首屏最多 4 到 5 个业务区块
3. 首屏不得出现第二张大表
4. 历史、日志、附录必须下沉
5. 用户不滚动也应看懂页面主任务

### 13.3 状态反馈约束

所有关键按钮点击后必须立即反馈：

1. 已提交
2. 运行中
3. 已完成
4. 失败

运行中状态必须明确展示：

1. 正在执行什么
2. 当前处理范围
3. 当前时间范围
4. 当前步骤
5. 下一步是什么

### 13.4 等待状态约束

不允许出现无上下文的“继续等待”或单纯转圈。

等待必须明确说明：

1. 在等什么
2. 预计还会发生什么
3. 用户此刻是否需要操作

### 13.5 空态与异常态约束

空态和异常态必须：

1. 只描述当前事实
2. 给出下一步动作
3. 不用说明文填充版面

## 14. 页面命名与模块命名规范

### 14.1 页面命名

正式页面名称采用动作型或结果型命名：

1. 盘后决策
2. 调度中心
3. 执行对账
4. 复盘分析
5. 策略治理

### 14.2 模块命名

模块名必须直接表达用途，例如：

1. 今日状态
2. 今日信号
3. 待处理事项
4. 当前持仓
5. 当前运行
6. 最近结果
7. 待回写成交
8. 异常对账
9. 复盘结论
10. 核心证据
11. 治理结论

不推荐继续使用以下模糊命名：

1. 概览
2. 信息
3. 综合视图
4. 数据面板
5. 说明
6. 辅助区

## 15. 当前实现与目标态的核心偏差

结合当前实现，主要偏差包括：

1. 导航仍更接近系统模块拆分，而不是投资工作流拆分
2. 默认首页仍承载过多信息，主线不够单一
3. 部分关键页面仍在用系统术语命名，而非用户任务命名
4. 首屏仍存在日志、异常、大表过重的问题
5. “策略是否失效”尚未在提醒、解释、治理三个层次上清晰分层
6. 页面仍部分依赖文案说明，而不是靠结构自解释

---

## 16. 落地顺序建议

### 16.1 第一阶段：先定结构

1. 固定 5 页导航
2. 明确每页职责边界
3. 统一命名

### 16.2 第二阶段：重做入口

1. 先重构 `盘后决策`
2. 再重构 `调度中心`
3. 再重构 `执行对账`

### 16.3 第三阶段：补解释层

1. 落成 `复盘分析`
2. 明确异常、止损、策略失效的证据链

### 16.4 第四阶段：补治理层

1. 将 `影子对比` 升级为 `策略治理`
2. 完成 keep / observe / replace / disable 决策链

### 16.5 总原则

落地顺序必须遵守：

**先让用户每天会用，再让用户每周、每月用得更深。**

---

## 17. 一句话总结

量化分析模块的目标态，不是一个展示很多数据和功能的后台，而应是一套围绕低频投资节奏组织的工作台：
