# 量化模块操作台化页面规约 Design

## 背景

量化分析模块下多个页面长期存在同类问题：

1. 首屏采用 hero、展示卡、大标题、大留白骨架，视觉上像 PPT 或汇报页，而不是可执行的操作台。
2. 页面经常把同一类信息重复展示在多个区块里，首屏占用很大面积，但有效信息很少。
3. 无数据、无阻断、无任务时，空态仍占据整屏，用户无法快速继续主线工作。
4. 页面整改缺少统一规约，导致每次任务只能局部修补文案或样式，难以真正解决结构性问题。
5. 验证流程长期停留在 build、接口、marker、登录层面，没有以真实登录后的首屏结果作为最终验收标准。

本规约用于约束 `ruoyi-ui/src/views/quant` 下所有页面，作为后续设计、开发、验收的统一基线。

## 适用范围

适用于量化分析模块全部页面：

- `dashboard`
- `jobs`
- `dispatch-manual`
- `dispatch-auto`
- `dispatch-detail`
- `operations`
- `execution`
- `review`
- `shadow`
- `symbols`
- `backtest`

以及这些页面直接依赖的首屏组件、状态构建器、空态组件、操作面板组件。

## 最高优先级原则

### P0 原则

量化模块统一成操作台骨架，所有 PPT、hero、大卡片骨架一律禁止。

### 说明

页面的重要性不能通过大面积留白、大圆角、大标题、装饰卡、展示型色块来表达，只能通过：

1. 当前业务事实
2. 当前关键动作
3. 当前阻断或风险
4. 当前结果与结论

## 页面设计规约

### 1. 页面骨架规约

所有量化模块页面首屏统一采用三层操作台骨架：

1. 紧凑页头
2. 主操作 / 主事实区
3. 次级折叠区 / 列表区 / 表格区

约束如下：

1. 紧凑页头只负责标题、状态标签、1 到 2 个关键动作。
2. 页头不得承载大段说明、口号、背景介绍、展示型摘要块。
3. 主操作 / 主事实区必须直接回答当前页最关键问题。
4. 次级内容统一下沉，包括历史、日志、详细定义、兼容工具、扩展图表、附加解释。
5. 首屏默认不超过 2 个主块；同类事实优先合并成表格或清单。
6. 空态必须收缩为一行状态条或一个很小的摘要块，不能主导首屏。

### 2. 硬性禁令

以下规则为量化模块首屏硬性禁令：

1. 禁止 hero 区。
2. 禁止单卡 `min-height`。
3. 禁止 `el-empty` 插画空态。
4. 禁止一块只放 1 到 2 行字却占整块卡。
5. 禁止重复表达同一信息。
6. 禁止“左大标题 + 右摘要卡”的汇报式骨架。
7. 禁止装饰性渐变、氛围背景、发光块来撑页面。
8. 禁止把解释文案作为主区主体。
9. 禁止无业务价值的占位块。
10. 禁止未分页的大列表占首屏。
11. 禁止用大字号、大留白制造“重要感”。
12. 禁止将历史、日志、兼容工具、高级工具放在首屏主视觉中心。
13. 禁止把“建议动作 / 可以继续 / 下一步”这类抽象口号单独做成大块展示卡。

### 3. 例外规则

允许的例外只有以下几类：

1. 页面标题可以保留，但只能是紧凑页头。
2. 状态标签可以保留，但只能贴近事实对象出现。
3. 回测研究、影子对比、复盘证据类页面可以在首屏出现图表，但图表必须直接服务当前判断。
4. 多步骤任务页面可以展示步骤信息，但必须是紧凑步骤行或事实表，不得做成流程海报。
5. 危险恢复动作可以单独显示动作区，但动作区必须紧凑，不得海报化。
6. 空态允许存在，但面积不得超过首屏高度的 20%。
7. 次级区可使用折叠面板，但默认收起。

### 4. 页面类型分型规约

#### 4.1 调度类页面

适用：

- 调度中心
- 手工调度
- 自动调度
- 调度详情

首屏必须优先回答：

1. 现在运行什么
2. 当前范围
3. 时间范围
4. 当前步骤
5. 下一步
6. 当前结果或异常

历史、日志、分片、调度定义全部下沉。

#### 4.2 运维类页面

适用：

- 运维中心

首屏必须优先回答：

1. 有没有阻断
2. 影响是什么
3. 当前建议恢复动作是什么
4. 当前技术状态如何

如果没有阻断，首屏必须立即收缩，不允许空一整屏。

#### 4.3 执行闭环类页面

适用：

- 执行回写

首屏必须优先回答：

1. 当前最重要异常是什么
2. 优先级如何
3. 当前待闭环对象有哪些
4. 最应该执行的处理动作是什么

工具区不能盖过异常主线。

#### 4.4 复盘 / 治理类页面

适用：

- 复盘分析
- 影子对比
- 标的体系

首屏必须优先回答：

1. 当前结论是什么
2. 当前结论动作是什么
3. 当前核心证据是什么

大图表、大案例、规则说明、历史资料统一下沉。

#### 4.5 研究类页面

适用：

- 回测研究

首屏必须优先回答：

1. 当前研究配置是什么
2. 当前研究结果摘要是什么
3. 当前最值得看的研究信号是什么

图表可进首屏，但必须服务当前判断，而不是作为视觉装饰。

#### 4.6 看板类页面

适用：

- 量化看板

首屏必须优先回答：

1. 今天现在该看什么
2. 今天现在该做什么
3. 最重要的 1 到 3 个入口是什么

不允许把所有对象层和所有链路信息平铺到第一屏里。

### 5. 组件级规约

#### 5.1 卡片

1. 卡片是信息容器，不是展示容器。
2. 默认使用紧凑卡片，不允许大圆角、大内边距、大留白。
3. 没有内容时卡片必须收缩，不允许空卡。
4. 同类事实优先合并成一张卡内表格或清单。

#### 5.2 事实项

1. 事实项优先使用 `label + value`、二列表格、紧凑清单。
2. value 应尽量使用业务事实，不使用解释句子。
3. 同类事实尽量纵向排列，减少大宫格。

#### 5.3 表格

1. 表格是量化模块首选的一等展示容器。
2. 结果、历史、异常、对象、技术状态优先落表。
3. 所有长列表必须分页。
4. 表格空态只允许一行文字。

#### 5.4 空态

1. 禁止插画。
2. 禁止整屏“暂无数据”。
3. 必须附带明确业务结论或下一步。

#### 5.5 折叠区

1. 折叠区只放次级信息。
2. 默认收起。
3. 折叠标题必须是业务名词，不写空泛标题。

#### 5.6 按钮区

1. 每页首屏最多 1 个主按钮。
2. 次级按钮默认不超过 3 个。
3. 主按钮必须对应当前页唯一最重要动作。
4. 导航按钮不能占主视觉，除非它是唯一有效动作。

## 开发执行规约

### 1. 开工前

1. 明确本次改的是哪个页面、哪个首屏问题。
2. 写出该页面首屏必须回答的 3 到 5 个问题。
3. 改动前必须看真实页面，不允许只读代码。
4. 改动前必须保存首屏截图作为基线。

### 2. 实施中

1. 每次只改一个页面的首屏骨架。
2. 优先删结构，再补内容。
3. 先删 hero、空态大卡、重复卡、展示块，再决定保留什么。
4. 不允许用换文案代替结构整改。
5. 不允许用调颜色、圆角、间距来掩盖骨架问题。

### 3. 自测要求

1. 改完必须真实登录系统进入目标页面。
2. 不得停留在登录页、接口 200、build 通过这类验证层级。
3. 必须验证：
   - 首屏是否仍有大面积空白
   - 首屏是否先回答关键问题
   - 是否仍有重复信息
   - 空态是否仍占屏
   - 主按钮是否仍是正确动作
4. 必须在真实业务数据下观察页面。
5. 相关页面至少验证：
   - 有数据态
   - 无数据态
   - 异常 / 阻断态

### 4. Playwright 与自动化要求

1. 冒烟脚本只能证明“能访问、没崩、关键文案存在”。
2. 冒烟脚本不能作为页面体验验收证据。
3. 页面体验类问题必须至少产出一张真实登录后的首屏截图。
4. 没有真实截图，不得声称“问题已解决”。

### 5. 验收要求

每次页面任务必须提交：

1. 改前首屏截图
2. 改后首屏截图
3. 本页首屏现在回答了哪些问题
4. 删除了哪些浪费空间的区块
5. 哪些次级内容被下沉
6. 还剩什么问题没有解决

### 6. 失败处理

1. 连续两轮改动后问题仍存在，必须停止继续写代码。
2. 必须先复盘是改错页面、改错层级、验证失真还是规约缺失。
3. 没找到根因前，不允许继续扩散修改。

## 交付物规约

### 1. 必交付代码

1. 页面代码改动
2. 必要的状态层或组件层改动
3. 与本页相关的最小测试改动

### 2. 必交付证据

1. 改前首屏截图
2. 改后首屏截图
3. 真实登录后的页面访问证据
4. 实际运行的验证命令
5. 验证结果摘要

### 3. 必交付说明

1. 首屏回答了哪些关键问题
2. 删除了哪些 PPT 化结构
3. 哪些信息被下沉
4. 空态如何收缩
5. 还剩哪些问题未解决

### 4. 完成定义

量化模块页面任务只有同时满足以下条件才算完成：

1. 符合本设计规约
2. 符合开发执行规约
3. 有真实页面截图证据
4. 首屏不再是 PPT 骨架
5. 首屏没有大面积无信息空白
6. 首屏能直接回答当前页关键问题

## 页面整改检查清单

每次量化模块页面任务开始前和结束后都必须过一遍：

1. 这个页面首屏最关键的 3 到 5 个问题是什么？
2. 首屏有没有 hero？
3. 首屏有没有单卡 `min-height`？
4. 首屏有没有插画空态？
5. 首屏有没有重复信息？
6. 首屏有没有一大块只放 1 到 2 行字？
7. 没数据时，首屏是否已经收缩成一行或一个很小摘要块？
8. 历史、日志、兼容工具是否已下沉？
9. 是否只有 1 个主按钮？
10. 是否做了真实登录后的页面截图验证？

## 推荐落地顺序

后续整改建议按以下顺序执行：

1. 调度中心
2. 运维中心
3. 调度详情
4. 执行回写
5. 量化看板
6. 复盘分析
7. 影子对比
8. 标的体系
9. 回测研究
10. 手工调度 / 自动调度

原因：

1. 调度中心和运维中心当前首屏问题最严重，且最影响主操作链路。
2. 调度详情与执行回写直接承接操作结果，应优先修正。
3. 看板、复盘、治理、研究页在此之后按同一操作台规约统一收敛。
