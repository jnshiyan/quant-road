# 调度中心状态卡设计

## 背景

调度中心首页的“今日状态”当前存在两个问题：

1. 用户第一眼看不到“现在该做什么”。
2. “是否可继续使用”和“建议先处理”混在同一层文案里，容易产生歧义。

本次设计目标是把首页状态卡改成一个高掌控感、低歧义的决策入口，让用户在 3 秒内回答下面两个问题：

1. 我现在该做什么？
2. 我现在还能不能继续用系统？

## 设计目标

- 优先展示建议动作，其次展示是否可继续。
- 对 `WARNING` 场景保留用户自主权，不强制打断。
- 对 `BLOCKED` 场景明确禁止继续，不给模糊空间。
- 页面结构固定，避免用户每次重新学习。
- 后端字段语义稳定，避免前端继续从模糊文案里推断行为。

## 推荐方案

采用 `B` 方案：`行动 + 可继续性双层`

原因：

- 比单纯“行动优先”更适合技术型用户，既给动作，也给风险边界。
- 比自动切换主按钮更透明，用户不需要猜系统规则。
- 能同时承载“系统建议”和“用户掌控感”。

## 卡片结构

状态卡固定为 5 个信息块，顺序不变：

1. `建议动作`
2. `状态标签`
3. `现在能否继续`
4. `原因说明`
5. `主按钮 / 次按钮`

### 展示规则

- 主标题显示建议动作，例如：`去运维中心处理`、`等待当前任务完成`、`进入量化看板`
- 右上角状态标签显示：`可用 / 运行中 / 警告 / 阻断`
- 第二层单独显示继续性判断，例如：`可以继续查看`、`暂不建议重复提交`、`当前不可继续`
- 原因说明只解释风险来源和触发条件，不承担动作决策职责

## 按钮策略

`WARNING` 场景采用固定双按钮：

- 主按钮：`去处理问题`
- 次按钮：`继续进入看板`

`BLOCKED` 场景：

- 主按钮：`去运维中心处理`
- 次按钮：禁用或隐藏

推荐理由：

- 系统明确表达推荐动作，但不替用户做过度判断。
- 用户始终保留“继续看板”的主动权，掌控感更强。
- 不引入“有时主按钮去处理、有时主按钮去看板”的隐式规则。

## 状态映射

### OPERABLE

- 建议动作：`进入量化看板`
- 现在能否继续：`可以继续`
- 主按钮：`进入量化看板`
- 次按钮：`查看调度明细`

### RUNNING

- 建议动作：`等待当前任务完成`
- 现在能否继续：`暂不建议重复提交`
- 主按钮：`查看当前运行明细`
- 次按钮：`刷新状态`
- 必须展示等待对象，例如：`正在等待 run-strategy 完成`

### WARNING

- 建议动作：按具体风险生成，例如：`确认基础标的回退`、`核对缺失标的`
- 现在能否继续：`可以继续查看`
- 主按钮：`去处理问题`
- 次按钮：`继续进入看板`
- 必须展示处理时机，例如：`做最新决策前建议先处理`

### BLOCKED

- 建议动作：`先处理阻断问题`
- 现在能否继续：`当前不可继续`
- 主按钮：`去运维中心处理`
- 次按钮：禁用或隐藏
- 必须明确阻断点，例如：`sync-daily 失败`、`无活跃 worker`

## 文案原则

所有状态卡文案统一遵循三段式：

1. `现在该做什么`
2. `当前能不能继续`
3. `什么时候必须处理`

### 示例：基础标的回退

- 建议动作：`确认基础标的回退`
- 当前能否继续：`可以继续查看`
- 原因说明：`本次基础标信息来自库内回退。若你需要依赖最新基础标信息做范围、映射或调度决策，请先补跑 sync-basic。`

## 后端字段契约

为避免前端从旧字段中推断复杂语义，状态卡接口应稳定提供以下字段：

- `headlineAction`: 建议动作标题
- `statusCode`: `OPERABLE / RUNNING / WARNING / BLOCKED / EMPTY`
- `statusLabel`: 中文状态标签
- `canContinue`: 布尔值
- `continueLabel`: 继续性短句
- `reason`: 风险原因说明
- `urgency`: `now / before_decision / later`
- `primaryAction`: 主按钮对象
- `secondaryAction`: 次按钮对象，可为空

兼容策略：

- 在过渡期可以继续保留 `code / label / reason / suggestion`
- 前端状态卡优先消费新字段
- 新旧字段并存期间，前端不再从旧字段文案推断按钮行为

## 前端改造范围

- 调整调度中心首页 `TodayStatusCard`
- 在状态卡中引入双层信息结构和固定按钮区域
- 与现有 `nextAction`、`primaryAction`、`secondaryActions` 做职责划分，避免重复表达

原则：

- 状态卡负责“首页快速决策”
- 任务面板负责“当前任务操作入口”
- 当前运行明细负责“过程观察和技术细节”

## 错误处理与边界

- `EMPTY` 场景：展示 `待执行`，建议动作为 `提交执行任务`
- 缺少等待对象时，`RUNNING` 场景回退为 `正在等待当前任务完成`
- 后端无法提供次按钮时，前端在 `WARNING` 场景默认回退为 `继续进入看板`
- `BLOCKED` 场景绝不显示“继续进入看板”可点击态

## 验证要求

### 后端

- 验证 `WARNING` 场景下建议动作、继续性和处理时机能同时返回
- 验证 `BLOCKED` 场景不返回可继续的次按钮
- 验证 `RUNNING` 场景带等待对象

### 前端

- 验证四类状态的卡片文案和按钮行为
- 验证 `WARNING` 状态固定显示双按钮
- 验证 `BLOCKED` 状态次按钮不可点击

## 非目标

- 本轮不重做整个调度中心首页布局
- 本轮不改动当前运行明细组件的结构
- 本轮不引入新的自动判定模式或隐藏式规则

## 决策结论

调度中心首页状态卡采用：

- 方案：`B 行动 + 可继续性双层`
- 交互：`WARNING` 固定双按钮
- 表达：统一三段式文案
- 实现：后端提供明确字段，前端停止从模糊文案推断逻辑
