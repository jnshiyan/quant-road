# 量化任务中心重设计方案

## 背景

当前量化任务页同时承担了值班、主流程执行、技术排障、兼容入口、日志输出等多类职责，导致以下问题：

- 用户进入首页后无法一眼判断今天是否可继续运营。
- 页面无法明确回答“系统现在在做什么、卡在哪里、下一步该做什么”。
- 动作区是按钮墙，按钮语义不清，点击前无法预期会发生什么。
- 全局 loading 让用户误以为所有动作都在运行，缺乏掌控感。
- 运行中状态只有抽象状态，没有“正在等待什么”的可观测反馈。
- 首页同时暴露运维和兼容入口，稀释了业务主路径。

本方案目标是将量化任务页重构为符合值班和指挥场景的产品界面：用户一眼知道系统状态、当前主任务、下一步建议、点击预期结果，并将技术恢复能力迁移到独立运维页。

## 设计目标

### 用户目标

- 一眼知道今天能不能继续推进业务。
- 一眼知道当前主任务跑到哪一步、在等什么、是否需要人工介入。
- 一眼知道现在该点哪个动作，以及点击后会发生什么。
- 在运行中拥有掌控感，能看到最近进展和必要的技术摘要。
- 在系统阻断或告警时，被明确引导到恢复路径，而不是被迫自己猜。

### 产品目标

- 首页只承担“推进业务”的职责，不再承担“恢复系统”的职责。
- 运维能力单独成页，不污染业务指挥首页。
- 标准路径与兼容兜底路径边界清晰。
- 所有主动作点击前必须有标准确认面板，避免黑盒执行。
- 所有运行中状态必须可观测，不能只给抽象状态词。

## 核心决策

### 1. 双页拆分

采用双页结构：

- **任务中心**：回答“今天能不能继续运营、当前主任务在干什么、我现在该做什么”。
- **运维中心**：回答“哪里坏了、影响多大、该怎么恢复、恢复后如何回到业务路径”。

兼容旧入口不再保留在首页，迁移到运维中心的高级工具箱。

### 2. 首页主视角

首页采用“B 指挥台优先 + C 页头总状态”的信息架构：

- 页头先展示 `今日状态`。
- 主体先展示 `当前主任务`。
- 之后给出 `下一步建议`。
- 再展示 `动作执行器` 和 `执行反馈`。

### 3. 今日状态语义

`今日状态` 是业务层判断，不是技术任务状态。仅回答“今天能否继续推进业务动作”。

固定三档：

- `可营业`：关键结果已可消费，可进入执行、看板、复盘。
- `警告`：仍可继续，但默认保守，建议先处理警告项。
- `阻断`：不允许继续执行或复盘，必须先进入恢复路径。

每次展示必须附带：

- 判定原因
- 当前建议

### 4. 首页只突出一个主任务

首页默认只展示 **1 个主任务**，由系统自动判定：

1. 阻断中的今日主流程
2. 仍在运行的今日主流程
3. 最近一次刚完成、且影响今日状态的主流程
4. 若今日无相关主流程，则显示“当前无主任务”

### 5. 主按钮位置固定、语义动态

首页主按钮固定在同一位置，但根据状态动态变化文案和动作：

- 可营业：进入下一步业务动作
- 警告：先处理警告
- 阻断：去运维中心处理

### 6. 主动作提交前必须确认

所有主动作在真正执行前弹出标准确认面板，固定展示：

- 本次动作的用意
- 将执行的步骤
- 执行范围
- 执行方式（同步 / 异步）
- 预期结果与去向

### 7. 可观测等待

运行中状态不能只显示“继续等待”，必须明确：

- 正在等待什么
- 系统现在在做什么
- 最近有哪些进展
- 什么情况下不应该继续等，而是应该人工介入

首页展示两层反馈：

- 业务层：当前阶段、预计结果、预计时长、是否需介入
- 技术摘要层：jobId、mode、step、shard、最近心跳

完整实时日志下钻到运维中心。

## 页面职责

### 任务中心

### 页面使命

只回答四个问题：

1. 今天能不能继续运营
2. 当前主任务跑到哪一步
3. 我现在该点哪个动作
4. 点击后会有什么预期结果

### 任务中心首页模块

#### 1. 今日状态

字段：

- 今日状态
- 判定原因
- 当前建议

行为：

- `警告` 时默认保守，不将继续业务作为主按钮。
- `阻断` 时首页主按钮直接导向运维中心。

#### 2. 当前主任务

字段：

- 任务名称
- 当前阶段
- 当前状态
- 正在等待什么
- 预计结果
- 是否需要人工介入

运行中额外字段：

- 已完成步骤 / 总步骤
- 当前范围
- 最近进展时间

#### 3. 运行反馈

默认展示最近 5 条关键进展，例如：

- 已提交任务
- 开始执行某步骤
- 某步骤完成
- 当前正在处理哪个分片或范围

同时展示技术摘要：

- jobId
- 执行模式
- 当前步骤
- 当前分片
- 最近心跳或最近进展时间

#### 4. 下一步建议

首页只允许 **1 条主建议**。建议类型包括：

- 继续等待
- 去执行页
- 去复盘页
- 去运维中心处理

建议必须附带原因，不能只给命令式文案。

#### 5. 动作执行器

首页最多显示 3 个可见动作：

- 1 个主按钮
- 2 个次按钮

动作类型规则：

- 主按钮：系统当前最推荐动作
- 次按钮 1：查看结果或进入下一业务页
- 次按钮 2：处理异常或进入运维中心

更多动作迁移到二级入口，不在首页平铺。

#### 6. 提交后反馈与时间线

点击动作后必须立即反馈：

- 正在提交什么
- 提交开始时间
- 提交是否成功
- 生成的 jobId
- 同步 / 异步
- 当前时间线
- 下一步结果去哪看

失败时必须给出：

- 失败原因
- 下一步建议
- 是否允许重试

### 运维中心

### 页面使命

只回答四个问题：

1. 哪里坏了
2. 影响有多大
3. 该怎么恢复
4. 恢复后如何回到业务路径

### 运维中心模块

#### 1. 阻断总览

字段：

- 当前最严重问题
- 问题层级（worker / 分片 / 批次 / 数据）
- 影响范围
- 主恢复动作

#### 2. 任务恢复

字段：

- 失败批次
- 失败步骤
- 最近失败信息
- 可恢复动作

动作：

- 重试失败批次
- 恢复失败步骤

#### 3. 异步执行器

字段：

- 活跃 worker
- 待消费分片
- 运行中分片
- 过期分片
- 最近心跳 / 租约

动作：

- 消费分片
- 恢复过期分片

#### 4. 数据完整性

字段：

- 缺失数据项
- 影响范围
- 是否阻断业务
- 建议补救动作

#### 5. 高级工具箱

内容：

- 兼容旧入口
- 手工消费
- 兜底修复动作

规则：

- 明确标记为“仅在标准路径失效时使用”
- 不出现在任务中心首页

## 任务中心与运维中心之间的跳转规则

采用“默认不强跳，只做强提醒”的策略：

### 可营业

- 不跳转
- 首页主按钮仍为业务推进动作

### 警告

- 不强跳
- 首页主按钮变为 `先处理警告`
- 可保留一个次按钮 `仍继续业务`，但必须明确风险

### 阻断

- 不自动跳页
- 首页主按钮变为 `去运维中心处理`
- 不再提供继续执行类主按钮

### 运行中但正常

- 不跳转
- 首页建议为 `继续等待`
- 同时提供：
  - 查看实时进展
  - 查看只读结果页

### 运行中但长时间无进展

- 首页主建议切换为 `去运维中心排查`
- 必须说明触发原因，例如：
  - 已超过阈值无新进展
  - worker 心跳中断
  - 分片长期未完成

## 文案原则

当前页面的说明文案存在“像说明书、但不解决问题”的问题。重设计后文案必须遵守以下原则：

- 先回答用户关心的问题，再补背景。
- 用人话解释系统状态，不直接堆抽象状态词。
- 说明文案必须能指导动作，不做空泛标语。
- “继续等待”必须说明等待对象。
- “可营业 / 警告 / 阻断”必须附带原因与建议。

禁止继续出现类似文案：

- `当前任务状态 SUCCESS，先确认主任务是否仍在推进。`
- `当前批次已业务就绪，可进入看板、执行或复盘页继续运营。`
- `当前后台执行器状态可用，排障重点转向失败批次和异常步骤。`

## 数据与接口要求

为支持本设计，首页至少需要以下聚合能力：

- 今日状态聚合结果
- 主任务自动选择结果
- 主任务当前阶段与等待对象
- 最近关键进展事件流
- 技术摘要（jobId / mode / step / shard / heartbeat）
- 下一步建议结果
- 主动作确认面板所需步骤、范围、方式、预期结果

运维中心至少需要以下聚合能力：

- 阻断总览聚合
- 失败批次与失败步骤
- worker / shard 运行摘要
- 数据完整性问题摘要
- 高级工具箱入口描述

## 错误处理

### 首页聚合接口失败

- 页面不应整体空白
- 保留上次成功快照（如果有）
- 显示“状态暂时不可刷新，请进入运维中心查看详细状态”

### 任务提交失败

- 立即显示失败原因
- 给出下一步建议
- 若是数据完整性阻断，直接引导去运维中心

### 日志流不可用

- 首页退化为“最近关键进展 + 最后更新时间”
- 运维中心标记“实时日志不可用，但任务状态仍可查询”

## 验证方案

### 体验验证

- 用户进入首页 3 秒内能回答“今天能不能继续运营”
- 用户进入首页 5 秒内能回答“当前主任务在做什么”
- 用户点击主按钮前能说出点击预期结果
- 用户看到“继续等待”时能说出“我在等什么”

### 功能验证

- 今日状态三档正确切换
- 主任务自动选择规则正确
- 首页主按钮语义随状态正确变化
- 标准确认面板字段完整
- 提交后能立即看到 jobId、模式和时间线
- 运维中心能承接阻断恢复路径

### 回归验证

- 任务中心 smoke 需要验证：
  - 首页状态展示
  - 主按钮确认面板
  - execute 提交链路
  - 提交后反馈
- 运维中心 smoke 需要验证：
  - worker 摘要
  - 失败批次
  - 过期分片恢复
  - 兼容入口可见但不在首页出现

## 非目标

以下内容不在本次首页重设计范围内：

- 重写量化执行后端调度模型
- 彻底移除所有兼容入口
- 重做看板 / 执行页 / 复盘页的业务内容
- 将首页做成完整日志控制台

## 推荐落地方向

本次实现建议优先完成以下顺序：

1. 拆出运维中心页面
2. 任务中心首页按新结构收敛
3. 接入主任务自动选择与今日状态聚合
4. 接入运行反馈与关键进展流
5. 接入标准确认面板
6. 调整 smoke 与回归测试
